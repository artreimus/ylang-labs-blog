DeepSWE matters because it makes frontier coding agents look less solved.

The top model on Datacurve's leaderboard scores 70% +/-4 on original, long-horizon engineering tasks. The next tier is in the mid-50s, and many models fall below 30%.

That is a healthier signal than another saturated benchmark where every frontier model lands inside the same error band.

<DeepSWELeaderboard />

That is why the benchmark spread through engineering AI Twitter. Kol Tregaskes pointed out that DeepSWE matches the developer intuition that GPT-5.5 has been stronger for coding than SWE-Bench near-parity suggested. Garry Tan called it "the new standard for engineering evals." Matthew Berman made the same practical point: DeepSWE reflects what he hears from engineers better than other benchmarks.

DeepSWE is not measuring autocomplete or recall. It asks whether an agent can enter an unfamiliar repository, find the right surface area, make a correct change, and leave nearby behavior intact.

## DeepSWE tests the part of coding that benchmarks usually sand down

Datacurve describes DeepSWE as 113 original tasks drawn from 91 active open-source repositories across TypeScript, Go, Python, JavaScript, and Rust. The tasks are intentionally new, not adapted from merged commits or public pull requests, which reduces the chance that the model is being rewarded for memorizing a public solution.

The construction choice is the whole story. Most software work is not "implement the function below." It is "make this behavior true in a codebase you did not write, without breaking everything nearby."

DeepSWE also makes the prompts shorter than SWE-Bench Pro while making the solutions larger. Datacurve reports that DeepSWE prompts are roughly half the length of SWE-Bench Pro's, while solutions require 5.5x more code and about 2x more output tokens.

## The leaderboard is interesting because the gaps are ugly

The current leaderboard has GPT-5.5 at 70% +/-4, GPT-5.4 at 56% +/-5, and Claude Opus 4.7 at 54% +/-5. After that, the drop is steep: Claude Sonnet 4.6 reaches 32% +/-4, Gemini 3.5 Flash reaches 28% +/-4, and several recognizable models sit much lower.

The optimistic read is that a 70% pass rate on original long-horizon repository tasks is a real achievement. The sober read is that 70% still means the best public configuration fails about three out of ten tasks. For production, that is not "replace the maintainer." It is "give the maintainer a powerful junior engineer who needs review, tests, and rollback paths."

The shape matters more than the winner. A benchmark that separates models by 10, 20, or 40 points is telling us something. A benchmark where everyone clusters together may have stopped matching the frontier.

## The verifier is the real product

The GitHub repository describes each task as a package: metadata, the instruction the agent sees, a reproducible environment, grading tests, and a held-out reference solution. The reference solution is not used at grading time. The verifier checks observable behavior, not whether the agent rediscovered the author's patch shape.

That matters because real engineering tasks admit many valid implementations. If a benchmark only accepts one helper name, file split, or control-flow shape, it is measuring whether the model guessed the author's implementation. DeepSWE tries to avoid that trap by grading behavior.

## The fixed harness is both a strength and a warning label

All leaderboard runs use mini-swe-agent. That holds the scaffolding constant: every model gets the same broad tool surface and agent loop instead of each vendor's native editor, patch primitive, and system prompt.

It also means the score is not a direct answer to "which coding product should my team buy?" Developers use Codex, Claude Code, Cursor, Gemini CLI, or internal frameworks with retrieval, code navigation, test selection, and review loops.

DeepSWE's fixed harness isolates model capability, while leaving room for model-native tooling to change the production result.

## Use DeepSWE as an eval design pattern

The lesson for builders is not "copy DeepSWE's leaderboard into a procurement doc." The lesson is to copy the evaluation shape:

1. Write original tasks against your own codebase, not public issues with known fixes.
2. Keep prompts behavior-focused instead of over-specifying the implementation.
3. Grade observable behavior through public APIs, CLI output, persisted state, or end-to-end tests.
4. Run regression checks so the agent cannot satisfy the new behavior by breaking nearby behavior.
5. Hold the harness fixed for model signal, then use product-native harnesses for deployment signal.

Model evals and product evals answer different questions: which base model is stronger, and which full agent system your team can trust.

## Where DeepSWE can still mislead

The corpus focuses on active open-source repositories with at least 500 GitHub stars, mostly in TypeScript, Go, and Python. That keeps tasks realistic, but it does not guarantee transfer to private monorepos, unusual build systems, C++, Java, mobile, or data infrastructure work.

DeepSWE's tasks are original today, not forever. Public benchmarks have finite shelf lives once future training and fine-tuning pipelines can absorb their surface area.

## The practical takeaway

The best benchmark is the one that makes capable models look uneven in ways developers recognize. DeepSWE does that.

It shows that frontier coding agents can solve real repository tasks at rates that matter, and that the field is still far from boring reliability.

If your internal coding-agent eval has every model tied near the top, raise the bar. Shorter prompts, original tasks, broader repositories, behavior-first verifiers, and regression checks will tell you more than another leaderboard clone.

That is the useful thing DeepSWE gives us: not just a ranking, but a better shape for asking whether an agent can actually do engineering work.
